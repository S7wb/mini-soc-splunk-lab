# SSH Successful Login Followed by Privilege Escalation — Alert Configuration

## Overview

This document describes the Splunk alert configuration used to detect a successful SSH login followed by sudo command execution and the opening of a root session for the same Linux account.

The alert represents a high-risk sequence in which a remotely authenticated account obtains root-level execution.

## Alert Details

| Setting | Value |
|---|---|
| Alert Name | `SSH Successful Login Followed by Privilege Escalation` |
| Description | Detects a successful SSH login followed by sudo execution and a root session for the same account |
| Alert Type | Scheduled |
| Schedule | Every 5 minutes |
| Cron Expression | `*/5 * * * *` |
| Search Time Range | Last 15 minutes |
| Internal Correlation Limit | 600 seconds |
| Trigger Condition | Number of results is greater than `0` |
| Trigger Mode | Once |
| Trigger Action | Add to Triggered Alerts |
| Severity | Critical |
| Display Mode | Digest |
| Throttle | Enabled |
| Throttle Duration | 15 minutes |
| Permissions | Private |

## Detection Query

The alert uses the validated SPL query stored in:

```text
detections/ssh-login-followed-by-privilege-escalation.spl
```

The query:

1. Searches for successful SSH authentication events.
2. Searches for sudo command execution.
3. Searches for root-session creation.
4. Classifies each event by activity type.
5. Extracts the SSH username and source IP.
6. Extracts the sudo username and executed command.
7. Extracts the account associated with the root session.
8. Sorts events chronologically.
9. Correlates each root session with the most recent SSH login and sudo command.
10. Requires the sequence to complete within ten minutes.
11. Returns each detected privilege-escalation sequence separately.

## Alert Workflow

1. A successful SSH password login is recorded in `/var/log/auth.log`.
2. Splunk Universal Forwarder sends the event to Splunk Enterprise.
3. The authenticated user executes a command using sudo.
4. A root session is opened for the same account.
5. The scheduled alert runs every five minutes.
6. The search reviews the previous fifteen minutes.
7. The SPL query confirms that the correlated sequence occurred within ten minutes.
8. Splunk creates a Critical alert.
9. The analyst reviews the source IP, username, command, and timestamps.
10. Throttling suppresses duplicate alerts generated by overlapping search windows.

## Negative Validation Tests

### SSH Login Without Privilege Escalation

A successful SSH login was generated without executing sudo or opening a root session.

The raw SSH event was present in Splunk, but the detection query returned:

```text
No results found
```

### Local Sudo Without SSH Login

The test account executed the authorized sudo command through a local session without a preceding SSH login inside the isolated test window.

The sudo activity was present in Splunk, but the detection query returned:

```text
No results found
```

These tests confirmed that both the SSH login and privilege-escalation activity are required.

## Why the Search Range Is 15 Minutes

The alert reviews the previous fifteen minutes even though the detection requires the escalation sequence to occur within ten minutes.

This overlap reduces the chance of losing the original SSH login event at the boundary of a scheduled search.

The SPL query still enforces:

```text
Root session time - SSH login time <= 600 seconds
```

Therefore, reviewing fifteen minutes of logs does not allow escalation sequences longer than ten minutes.

## Throttle Configuration

The alert initially generated repeated notifications because:

- It ran every five minutes.
- It searched the previous fifteen minutes.
- The same matching sequence remained inside multiple overlapping windows.

A fifteen-minute throttle was enabled.

After enabling throttling:

1. A new Critical alert triggered successfully.
2. The following scheduled execution did not create a duplicate.
3. Only one alert remained visible for the retained activity.

Because the alert uses `Trigger Once` and `Digest` mode, throttling applies to the alert as a whole rather than individual result fields.

## Why Critical Severity Was Selected

Critical severity was selected because the detection confirms:

```text
Remote Authentication
→ Privileged Command Execution
→ Root Session
```

This sequence indicates that a remotely authenticated account obtained root-level execution.

In a production environment, it may represent:

- Compromised credentials
- Unauthorized administrative access
- Successful privilege escalation
- Preparation for persistence
- Defense evasion
- Credential access
- Lateral movement
- Sensitive data access

The severity may be reduced after confirming that the activity was authorized administrative work.

## Recommended Production Tuning

| Setting | Tuning Consideration |
|---|---|
| Search range | Preserve overlap with the scheduled execution frequency |
| Correlation limit | Adjust based on expected administrative behavior |
| Severity | Increase priority for sensitive assets and privileged accounts |
| Throttling | Adjust based on duplicate-alert volume |
| Allowlisting | Exclude approved administrative systems only when justified |
| Account context | Enrich results with account privilege and ownership |
| Command context | Increase risk for suspicious or uncommon commands |
| Source enrichment | Add IP reputation, GeoIP, and network-zone context |
| Audit telemetry | Add `auditd` or process events for stronger session correlation |
