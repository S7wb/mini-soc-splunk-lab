# Lessons Learned

## Overview

This document summarizes the technical and operational lessons learned while building and validating the Mini SOC lab with Splunk Enterprise.

The project provided hands-on experience in log collection, SPL development, alert configuration, controlled attack simulation, investigation, troubleshooting, and incident documentation.

## 1. Log Collection Must Be Verified First

A detection cannot work correctly unless the required logs are being collected and indexed.

Before troubleshooting the SPL query, verify:

- The monitored log file exists
- Splunk Universal Forwarder is running
- The forwarder is connected to the Splunk Server
- Events are reaching the correct index
- The expected host and source values are present
- Recent events are visible in Splunk

The following search helped confirm that events from the victim machine were reaching Splunk:

```spl
index=main host=victim source="/var/log/auth.log"
| sort 0 - _indextime
| table _time _indextime _raw
| head 20
```

## 2. Current IP Addresses Must Be Confirmed Before Testing

Virtual machine IP addresses may change after restarting the lab.

During testing, the victim machine address had changed from a previously used address to:

```text
192.168.56.20
```

The initial Hydra test was sent to the outdated address, so no corresponding SSH events appeared under the victim host.

Running the following command before every test prevents this issue:

```bash
hostname -I
```

The current destination address should always be confirmed before running Hydra or troubleshooting missing events.

## 3. Raw Logs Should Be Reviewed Before Writing SPL

Reviewing the raw authentication events revealed that the lab contained different authentication event formats.

Examples included:

```text
Failed password for saeed from 192.168.56.30
```

```text
Failed password for invalid user testuser from 192.168.56.1
```

```text
authentication failure ... rhost=192.168.56.30 user=saeed
```

There were also local desktop authentication failures generated by `gdm-password` with an empty `rhost` value.

The final SSH detection was therefore based specifically on `Failed password` events generated by `sshd`, preventing unrelated local authentication failures from entering the result.

## 4. One Authentication Attempt Can Produce Multiple Events

A single failed SSH login may generate both:

- `authentication failure`
- `Failed password`

Counting both event types could inflate the number of authentication attempts.

The validated detection uses `Failed password` events as the primary record for counting failed SSH passwords.

This reduced the risk of counting the same authentication attempt more than once.

## 5. Compressed Linux Events Must Be Counted Correctly

Linux may compress repeated authentication messages into an event such as:

```text
message repeated 5 times: [ Failed password for saeed from 192.168.56.30 ... ]
```

This event represents five attempts, not one.

The SPL detection extracts the repeated count and calculates the logical number of attempts:

```spl
| eval attempts=coalesce(tonumber(repeat_count), 1)
```

During validation, Splunk displayed three raw events while correctly calculating six failed authentication attempts.

## 6. Field Extraction Should Match the Actual Log Format

The source IP address and username were stored inside `_raw` and required extraction with `rex`.

The final extraction supports:

- Standard usernames
- Invalid-user SSH attempts
- IPv4 source addresses
- Compressed `message repeated` events

Useful extracted and calculated fields included:

- `host`
- `src_ip`
- `targeted_users`
- `failed_attempts`
- `first_seen`
- `last_seen`
- `duration_seconds`

These fields made the detection result immediately useful for alert triage.

## 7. Detection Thresholds and Time Windows Must Work Together

The validated detection threshold was:

```text
5 or more failed SSH attempts
```

The validated alert search window was:

```text
Last 5 minutes
```

The alert schedule was configured to run every five minutes:

```text
*/5 * * * *
```

Using a matching search window and schedule provides consistent coverage while avoiding unnecessary overlap.

The threshold may require tuning in another environment based on normal authentication behavior.

## 8. Historical Searches and Scheduled Alerts Serve Different Purposes

For historical analysis, five-minute buckets can be created with:

```spl
| bin _time span=5m
```

However, fixed buckets may split activity that occurs across a bucket boundary.

For the scheduled alert, the query does not use `bin`. Splunk instead runs the search over the previous five minutes according to the alert configuration.

This allows the alert to evaluate the complete rolling search window at execution time.

## 9. A Query Must Be Tested Manually Before Becoming an Alert

The detection was first tested manually in Splunk.

The validated result showed:

| Field | Result |
|---|---|
| Destination host | `victim` |
| Source IP | `192.168.56.30` |
| Targeted user | `saeed` |
| Failed attempts | `6` |
| Activity duration | Approximately `14.78` seconds |

Only after confirming these results was the query added to the scheduled alert.

This reduced the chance of troubleshooting query logic and alert scheduling at the same time.

## 10. Scheduled Alerts Must Also Be Tested on Their Actual Schedule

A successful manual search does not prove that the alert configuration works.

The alert was configured with:

| Setting | Value |
|---|---|
| Alert type | Scheduled |
| Schedule | Every 5 minutes |
| Search window | Last 5 minutes |
| Trigger condition | Number of results greater than `0` |
| Trigger mode | Once |
| Severity | Medium |

After controlled Hydra activity, the alert appeared in the Splunk Triggered Alerts page at the next scheduled execution.

The triggered result contained the expected detection fields and confirmed that both the query and the alert configuration were working.

## 11. Alert Severity Should Reflect the Available Evidence

The failed SSH brute-force alert was assigned Medium severity because it confirmed repeated failed authentication attempts but did not confirm successful access.

Severity should be increased when additional evidence is observed, such as:

- A successful login after repeated failures
- Authentication to a privileged account
- Suspicious commands after login
- Privilege escalation
- Persistence activity
- Connections from a known malicious source

Failed attempts alone and successful compromise should not be treated as the same level of risk.

## 12. Security Controls Can Affect Detection Testing

During validation, Hydra displayed repeated connection errors because Fail2Ban blocked the source address after detecting the SSH activity.

Splunk also generated separate alerts related to the Fail2Ban action.

This demonstrated that preventive controls can affect attack simulations and should be considered during testing.

When several controls are active, the analyst should distinguish between:

- The original brute-force activity
- The detection alert
- The defensive blocking action
- Additional alerts generated by the response control

## 13. Dashboards Support Monitoring but Do Not Replace Investigation

Dashboards provide useful visibility into authentication activity and trends, but they do not provide the complete context of an incident.

The analyst still needs to review:

- Raw events
- Related successful authentications
- Activity before and after the alert
- Other systems targeted by the source IP
- Defensive actions such as Fail2Ban blocks
- Possible false-positive explanations

## 14. Incident Documentation Is Part of the SOC Workflow

Writing an incident report helped organize the investigation into:

- Incident summary
- Alert details
- Data source
- Detection logic
- Evidence
- Extracted indicators
- Investigation steps
- Assessment
- Recommended response actions
- Closure

The final report documented the actual validated evidence instead of using placeholder values.

## 15. Controlled Testing Is Essential

All Hydra activity was performed inside an isolated and authorized virtual lab.

Controlled testing made it possible to:

- Generate predictable events
- Compare attack output with Linux logs
- Validate field extraction
- Confirm the detection threshold
- Trigger the scheduled alert
- Observe the Fail2Ban response
- Document the complete SOC workflow safely

## 16. Future Learning Areas

The project identified several areas for future improvement:

- Detecting failures followed by a successful SSH login
- Monitoring privileged-account authentication
- Detecting distributed brute-force attacks
- Detecting low-and-slow password attacks
- Adding Windows event logs and Sysmon
- Mapping detections to MITRE ATT&CK
- Adding GeoIP enrichment
- Improving alert throttling and allowlisting
- Expanding automated response testing
- Developing additional web and authentication use cases

## Conclusion

The Mini SOC lab demonstrated that effective detection involves more than writing an SPL query.

A complete workflow requires reliable log ingestion, accurate field extraction, realistic counting logic, controlled testing, alert scheduling, investigation, severity assessment, defensive-control awareness, and clear documentation.
